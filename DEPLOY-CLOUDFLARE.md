# Deploying capa-language.com on Cloudflare

This site is a static site generated by `build.py` from the fragments in
`_bodies/`. It is being migrated from GitHub Pages to Cloudflare (Workers
Static Assets, deployed with `npx wrangler deploy`). This document is the
step-by-step migration, and why.

## Why move

The one real win is **preview deploys**. On GitHub Pages every push to `main`
goes straight to the public site with no staging step. On Cloudflare every
branch and pull request gets its own preview URL, built from that exact commit,
so you can see the real site before it is public. Merge to `main` promotes to
production. Secondary wins: a faster global CDN, real HTTP security headers
(via `_headers`), and cookie-free analytics if you want them.

## What is prepared in the repo

- **`build.py --dist`** builds the site into a `dist/` folder that contains
  ONLY what the site serves (the pages, `book/`, `capa_logo.svg`, `styles.css`,
  `menu.js`, `fonts/`, `sitemap.xml`, `robots.txt`, `_headers`). It is the mode
  Cloudflare must use. Plain `python build.py` still builds in place into the
  repo root, which is what GitHub Pages serves during the overlap.
- **`wrangler.jsonc`** sets `assets.directory` to `dist`, so `npx wrangler
  deploy` uploads only the built site. This is the fix for the initial deploy,
  which uploaded the entire repository (including `.git/` and the source) because
  the assets directory defaulted to the repo root.
- **`_headers`** (emitted into the output by `build.py`, reusing the single
  `CSP` constant) sets the Content-Security-Policy as a real HTTP header plus
  `X-Content-Type-Options`, `Referrer-Policy` and `Permissions-Policy`.
- **`.gitignore`** ignores `dist/` and `.wrangler/`; the site is rebuilt on
  every deploy rather than committed.
- `build.py` uses only the Python standard library, so the Cloudflare build
  needs no dependency install.

## Build settings (Cloudflare dashboard)

In the Worker's build configuration set:

| Setting | Value |
|---|---|
| Build command | `python build.py --dist` |
| Deploy command | `npx wrangler deploy` (default) |
| Environment variable | `PYTHON_VERSION = 3.11` |

The `--dist` on the build command is the important part: without it the build
writes to the repo root and the deploy would again serve `.git/` and the
source. `wrangler.jsonc` (committed) tells wrangler to upload `dist/`, so there
is no separate "output directory" to set.

## Automatic deployments (the Git connection)

Builds fire on push only while the project is connected to the Git account.
In Settings -> Build, the banner "This project is disconnected from your Git
account" means pushes will not trigger builds; reconnect via Manage (or
Disconnect and reconnect the repo). Reconnecting does not retroactively build
commits already pushed while it was disconnected, so trigger one build
manually (Deployments -> Create deployment / Retry, or push a new commit)
after reconnecting. Production branch is main, build watch paths are `*`.

## Cutover, step by step (no downtime)

Content is identical on both hosts during the overlap, so there is no bad
moment.

1. **Create the Cloudflare account** and add `capa-language.com` as a zone
   (Cloudflare scans and imports the existing DNS records). Cloudflare gives you
   two nameservers; do not change the registrar yet.
2. **Set the build command to `python build.py --dist`** in the Worker's build
   settings, and confirm `wrangler.jsonc` is committed (it is). Trigger a
   deploy. The `*.workers.dev` preview URL should now serve only the site, with
   no `/.git/...` or `/build.py` paths.
3. **Verify the preview URL fully** before touching DNS: every page loads, fonts
   render, the stylesheet applies, the book PDF link works, `sitemap.xml` and
   `robots.txt` resolve, `/.git/HEAD` returns 404, and
   `curl -I https://<preview>.workers.dev/` shows the `content-security-policy`
   header. Push a throwaway branch to confirm a per-branch preview URL is
   generated.
4. **Add the custom domain** to the Worker (Settings -> Domains and Routes ->
   add `capa-language.com`, and `www` if you want it). Cloudflare stages the DNS
   record in the zone, which is not authoritative yet, so nothing changes live.
5. **Flip the nameservers** at your registrar to the two Cloudflare
   nameservers. Propagation is usually minutes. Resolvers still pointing at
   GitHub Pages keep serving the identical site until they catch up.
6. **After Cloudflare is authoritative and serving** the custom domain over
   HTTPS (the certificate is issued automatically):
   - Remove the custom domain from the GitHub repo's Pages settings so the two
     hosts stop competing.
   - Once fully off GitHub Pages, you can `git rm` the generated `*.html` at the
     repo root, `CNAME`, and `.nojekyll` (they are the GitHub-Pages artefacts;
     the site now builds fresh into `dist/`). Leaving them is harmless too.

## Post-cutover verification checklist

- `curl -I https://capa-language.com/` returns `200` and includes
  `content-security-policy`, `x-content-type-options: nosniff`, and
  `referrer-policy`.
- `https://capa-language.com/.git/HEAD` returns `404` (the source is not
  served).
- A spot check of pages (`/`, `/reference.html`,
  `/learn/14-from-source-to-sbom.html`) renders with fonts and styles, no
  console CSP violations, no mixed content.
- `https://capa-language.com/book/Capa-The-Capability-Typed-Programming-Language.pdf`
  downloads.
- `https://capa-language.com/sitemap.xml` and `/robots.txt` resolve.

## Optional follow-ups

- **Keep the source out of search results.** `robots.txt` (emitted by
  `build.py`) can gain a `Disallow: /_bodies/` line, though with the `dist/`
  build the fragments are no longer served at all.
- **Real redirect for `manifest.html`.** It is currently a meta-refresh to
  `learn/14-from-source-to-sbom.html`. A `_redirects` file in `dist/`
  (`/manifest.html /learn/14-from-source-to-sbom.html 301`) makes it a proper
  301, which `build.py --dist` could emit alongside `_headers`.
